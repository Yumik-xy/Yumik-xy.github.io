这些工具解决的问题各不相同，按它们在开发流程中所处的层次来分类：

| 工具 | 层次 | 一句话介绍 |
|------|------|-----------|
| [GSD](#gsd) | 工作流编排 | 把大项目拆成小任务，每个任务派全新子代理执行 |
| [OpenSpec](#openspec) | 规格管理 | 用 Markdown 写变更提案，AI 按文档干活，你按文档验收 |
| [PUA](#pua) | 行为约束 | 逼 AI 穷尽所有方案再放弃，用大厂方法论规范做事过程 |

**工作流编排**管的是「项目怎么拆、怎么执行、怎么验收」；**规格管理**管的是「每次改动要改什么、改成什么样」；**行为约束**管的是「AI 做事的态度——该多努力、该多仔细」。三者可以独立使用，也可以叠加。

## GSD

[GitHub](https://github.com/gsd-build/get-shit-done) \| 工作流编排框架

AI 写代码有个绕不开的问题：对话越长，输出质量越差。上下文窗口塞满了之前的讨论、报错、修改记录，AI 的注意力被稀释，开始犯低级错误——这就是所谓的「上下文腐烂」。

GSD 的解决思路是：把一个大项目拆成多个阶段，每个阶段再拆成具体任务，每个任务 spawn 一个全新的子代理去执行。子代理拿到的是干净的上下文——只有这个任务需要的规格和代码，没有历史包袱。执行完毕后提交一个原子 git commit，下一个任务继续用全新上下文。

整个生命周期覆盖：`初始化 → 规划 → 执行 → 验证 → 交付`。

**常用命令**

| 命令 | 用途 |
|------|------|
| `/gsd:new-project` | 初始化项目，生成规格和路线图 |
| `/gsd:plan-phase N` | 规划第 N 阶段 |
| `/gsd:execute-phase N` | 执行第 N 阶段 |
| `/gsd:autonomous` | 全自动，所有阶段一口气跑完 |
| `/gsd:fast` | 小任务快速执行，跳过规划开销 |
| `/gsd:debug` | 系统化调试，状态跨会话持久 |
| `/gsd:resume-work` | 恢复上次中断的工作 |
| `/gsd:do` | 说人话，自动匹配对应命令 |

**实际使用场景**

新项目从零开始时，`/gsd:new-project` 会做一轮需求访谈，然后生成路线图和阶段划分。之后逐阶段 `plan-phase` + `execute-phase` 推进，每个阶段的产出都有独立的验证环节。如果只是一个小改动不想走完整流程，`/gsd:fast` 可以跳过规划直接执行。

对于已有项目加新功能，同样走 `plan-phase` + `execute-phase`，GSD 会基于已有代码做规划，不会从头来。`/gsd:debug` 适合跨会话的复杂调试——调试状态会持久化，下次打开可以从断点继续。

- **优点**：每个任务独立上下文，不会随项目推进质量退化；原子 git commit 可回滚；可配模型组合平衡成本
- **缺点**：重度绑定 Claude Code；60+ 命令上手成本高；每个阶段生成大量中间文件
- **适合**：大型多阶段项目、需要长时间持续开发的场景

## OpenSpec

[GitHub](https://github.com/Fission-AI/OpenSpec) \| 规格管理框架，支持 Claude Code、Cursor、Copilot 等 21+ AI 工具

在和 AI 聊天写代码的过程中，需求往往是口头描述的，改着改着就变了，回头看聊天记录也找不到当时到底定了什么。OpenSpec 要解决的就是这个问题——每次改动之前先写一份变更提案，把「要改什么、为什么改、改成什么样」落到 Markdown 文件里。

流程三步：`Propose → Apply → Archive`。

`Propose` 阶段会在 `openspec/changes/` 下生成一组文件：提案（改动原因和范围）、规格（需求和场景）、设计文档（技术方案）、任务清单。`Apply` 阶段让 AI 按任务清单逐项执行。`Archive` 阶段把完成的变更归档，更新主规格供后续参考。

用 Delta 标记（`ADDED` / `MODIFIED` / `REMOVED`）追踪每次变更涉及的内容，对在已有代码库上做增量修改特别友好。

**常用命令**

| 命令 | 用途 |
|------|------|
| `openspec init` | 初始化项目 |
| `/opsx:new <name>` | 创建新的变更提案 |
| `/opsx:ff` | 一键生成所有规划文档 |
| `/opsx:apply` | 按规格执行实现 |
| `/opsx:archive` | 归档完成的变更，合并到主规格 |

**实际使用场景**

适合在已有项目上做迭代。比如要给一个 API 加字段，先 `/opsx:new add-user-email` 创建提案，把涉及的接口、数据库、前端展示都写清楚，然后 `/opsx:apply` 让 AI 按清单执行。做完 `/opsx:archive` 归档，下次再改相关功能时可以翻归档看之前的决策。

*OpenSpec 只管规格不管执行——它不编排子代理、不做自动验证。如果需要完整的执行流程，可以和 GSD 搭配使用：OpenSpec 管规格，GSD 管执行。*

- **优点**：纯 Markdown 不锁定任何 AI 工具、完全本地不需要 API Key、Delta 标记对增量修改友好
- **缺点**：只管规格不管执行、生态较新社区资源少
- **适合**：多工具混用的团队、在已有代码库上做增量迭代、需要留痕可追溯的变更管理

## PUA

[GitHub](https://github.com/tanweai/pua) \| 行为约束插件

AI 写代码有个常见的问题：试了两三次没搞定，就开始说「建议你手动操作」或者「这可能是环境问题」。本质上是在归因于外部因素，而不是继续深挖。

PUA 要解决的就是这个——给 AI 注入一套行为约束，核心是三条规则：做完了要拿验证结果证明（不是嘴上说完了）、归因之前先用工具验证（不是猜）、穷尽所有方法之前不许说不行。

### 压力升级

插件有一套随失败次数递增的压力机制。第一次失败时只是建议换个方法。到第三第四次，要求 AI 必须搜索文档、读源码、列出至少三个不同的假设逐个验证。到第五第六次，进入强制清单模式——七项排查动作全部做完才允许继续。

这套机制的实际效果是：AI 不会在同一个思路上反复打转，每次失败都被推向更深层的排查。

### 方法论路由

插件会根据任务类型自动选择不同的做事方法。这里叫「味道」——不同大厂的文化风格，但重点不是语气，而是背后绑定的方法论：

- 修 bug 走华为味：强制做 RCA 根因分析，不允许头痛医头
- 写新功能走 Musk 味：先质疑需求是否合理，删掉不必要的部分，再简化方案
- 架构决策走 Amazon 味：从用户需求倒推，写清楚设计文档
- 性能优化走字节味：数据驱动，要有 before/after 的对比数据

不需要手动选，插件会根据当前任务自动路由。

### 冰山法则

另一个有价值的机制是「冰山法则」——修了一个 bug，AI 会被要求检查：同模块有没有同类问题？上下游有没有被波及？目标是「一个问题进来，一类问题出去」，而不是修完眼前这个就收工。

### 常用命令

| 命令 | 用途 |
|------|------|
| `/pua:pua` | 核心模式，自动选方法论 |
| `/pua:p7` | P7 骨干模式——方案先行，影响分析，三问自审 |
| `/pua:p9` | P9 管理者——不写代码，拆任务、写 Prompt、管子代理 |
| `/pua:p10` | P10 CTO——定方向、管 P9 |
| `/pua:pua-loop` | 自动迭代模式，最多 30 轮，直到任务完成 |
| `/pua:yes` | 鼓励模式——行为约束不变，语气换成正向激励 |
| `/pua:mama` | 妈妈唠叨模式——行为约束不变，语气换成碎碎念 |

### 实际使用场景

日常开发直接用 `/pua:pua` 就够了，插件会根据当前任务自动切换方法论。

遇到需要长时间无人值守的任务，`/pua:pua-loop` 会自动循环执行：验证→发现问题→修复→再验证，直到所有条件满足。遇到缺环境变量之类需要人介入的情况会自动暂停，不会卡死。

项目比较大、需要多个子代理并行时，`/pua:p9` 适合做任务拆解——P9 模式下 AI 只负责拆任务和写 Prompt，然后 spawn 多个 P8 代理去执行。个人独立开发一般用不到这个层级。

`/pua:yes` 和 `/pua:mama` 是旁白风格的切换，底层的行为约束和核心模式完全一样。前者是 ENFP 式的鼓励，后者是中国式妈妈的碎碎念，适合换个口味。

- **优点**：失败时自动触发、方法论路由让不同任务有不同的排查策略、冰山法则推动关联问题排查
- **缺点**：旁白输出会占用上下文窗口、味道和职级体系内容较多需要时间熟悉
- **适合**：AI 容易放弃的复杂调试、需要 AI 主动排查而不是被动等指令的场景
